Actions are the core building blocks of ServFlow workflows. They perform discrete operations such as fetching data from a database, calling external APIs, running AI agents, or transforming values. Every workflow consists of an entry point, one or more actions, and a response.

For detailed documentation on each action type, see the [Actions reference pages](/actions/overview).

---

## Working with Actions in the Dashboard

The ServFlow dashboard provides a visual interface for adding and configuring actions in your workflows.

### Adding an Action

To add an action to your workflow:

1. Click the **+** button below an existing node in the workflow canvas
2. Select **Actions** from the node type options
3. Browse the list of available actions in the sidebar

### Browsing Available Actions

When you click to add an action, a sidebar panel displays all available action types. Each action shows its name and a brief description of what it does.

### Configuring an Action

Selecting an action opens the configuration panel on the left side of the screen. Each action type has specific fields to configure.

<Frame>
  <img src="/images/concepts/action-config.png" alt="Action configuration panel showing AI Agent settings" />
</Frame>

The configuration panel displays:

- **Action type** â€” The selected action (e.g., AI Agent)
- **Configuration fields** â€” Type-specific settings such as identifiers, prompts, or connection details
- **Save Action** â€” Button to save your configuration

<Tip>
Use `{{ .Variable }}` syntax in input fields to reference dynamic content from previous actions or request parameters. See [Dynamic Content](/references/concepts/dynamic-content) for more details.
</Tip>

### Action Outputs

Actions have output handles that let you define the flow:

- **Next** â€” Where to route on successful completion
- **Fail** â€” Where to route when an error occurs
- **Tools** â€” Additional outputs for specific action types (e.g., AI Agent tools)

Drag connections from these handles to other actions, conditionals, or response nodes to build your workflow logic.

### Using Action Results

When an action completes, its result is stored and can be referenced by subsequent nodes. Use the `{{ .action_id }}` syntax in any input field to access these values.

For example, if an action with ID `get_user` returns user data, you can access fields in later actions:

- `{{ .get_user }}` â€” The full result object
- `{{ .get_user.email }}` â€” A specific field from the result
- `{{ .get_user.profile.name }}` â€” Nested field access

## Advanced: YAML Configuration

For code-based workflows, actions can be defined in YAML configuration files. This approach is useful for version control, CI/CD pipelines, or programmatic configuration generation.

### Action Structure

Every action follows this structure in YAML:

```yaml
actions:
  action_id:
    type: action_type
    config:
      # type-specific configuration
    next: action.next_step
    fail: response.error
```

The `action_id` is a unique identifier you choose for the action. You'll use this ID to reference the action's results in other parts of your workflow.

### Common Fields

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `type` | string | Yes | The action type identifier (e.g., `fetch`, `http`, `agent`) |
| `config` | object | No | Type-specific configuration options |
| `next` | string | No | Step to execute on success |
| `fail` | string | No | Step to execute on failure |

### Chaining Actions

Use the `next` and `fail` fields to define your workflow's execution path:

- **`next`** â€” Routes to another step when the action succeeds
- **`fail`** â€” Routes to an error handler when the action fails

The routing syntax follows the pattern `type.identifier`:

- `action.other_action` â€” Route to another action
- `conditional.check_value` â€” Route to a conditional branch
- `response.success` â€” Route to a response

### Accessing Action Results

Action results are stored under the action's ID and can be accessed using template syntax:

```yaml
# Access the full result from action 'fetch_user'
value: "{{ .fetch_user }}"

# Access a nested field
value: "{{ .fetch_user.email }}"

# Use in expressions
expression: "{{ gt (len .fetch_user) 0 }}"
```

Here's an example of chaining actions where the second action uses data from the first:

```yaml
actions:
  get_user:
    type: fetch
    config:
      integrationID: my_database
      table: users
      filters:
        - field: id
          operator: eq
          value: "{{ param \"user_id\" }}"
      single: true
    next: action.send_welcome

  send_welcome:
    type: email
    config:
      recipientEmail: "{{ .get_user.email }}"
      subject: "Welcome!"
      content: "Hello {{ .get_user.name }}, welcome to our platform."
    next: response.success
```

## Available Actions

ServFlow provides a comprehensive set of actions organized by category. Click through to the detailed reference pages for full configuration options and examples.

### Data Operations

| Action | Description |
|--------|-------------|
| [`fetch`](/actions/data-operations#fetch) | Retrieves data from a database table |
| [`store`](/actions/data-operations#store) | Stores a new record in a database table |
| [`update`](/actions/data-operations#update) | Updates existing records |
| [`delete`](/actions/data-operations#delete) | Deletes records from a database table |
| [`mongoquery`](/actions/data-operations#mongoquery) | Executes native MongoDB queries |

### Vector Operations

| Action | Description |
|--------|-------------|
| [`storevector`](/actions/vector-operations#storevector) | Stores vector embeddings |
| [`fetchvectors`](/actions/vector-operations#fetchvectors) | Queries vectors for similarity search |

|--------|-------------|
